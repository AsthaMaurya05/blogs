MY NOTES ON YOLO — v1 THROUGH v3

Every earlier detector looked at an image hundreds of times. YOLO looks once — and that one decision changed everything.

I went through Redmon et al.'s original "You Only Look Once" paper, worked through my own class notes on the grid math and loss function line by line, and rebuilt the whole v1 → v2 → v3 story from a set of teaching notes that walked me through why each version exists. This is that, written the way I actually want to remember it.

01 · The Problem Before The Solution

Why a normal CNN can't do detection

A plain image-classification CNN answers exactly one question: "what is the main thing in this image?" Feed it a photo, get back one label. That's it.

This works when there's one object, roughly centered, and you don't care where it is — just what it is. Real-world images break all three assumptions at once. There can be several objects in one frame, they can sit anywhere at any size, and for the applications that actually matter — self-driving cars, security cameras, robotics — knowing where something is matters as much as knowing what it is.

The core limitation A classification-only CNN has no mechanism to output position, and no mechanism to output multiple things at once. It can tell you "dog," never "dog, right here, and a person over there too."

The first fix people tried was the sliding window approach: run the classifier over hundreds of little patches cut from the full image. That evolved into smarter two-stage pipelines — the R-CNN family — which first propose regions likely to contain an object, then classify each region separately. Both approaches share the same fatal cost: the CNN runs hundreds or thousands of times per image. Accurate, but nowhere close to real time.

02 · You Only Look Once

Detection reframed as one regression problem

YOLO's entire idea fits in one sentence: instead of looking at the image hundreds of times, look at it once.

Pass the whole image through a single CNN, one time, and have the network directly output what objects are present, where they are, and it does this for every object at once — all in one forward pass. Detection stops being "classify this crop, then that crop, then that crop" and becomes one big regression problem: the network directly predicts numbers — box coordinates and class probabilities — instead of running classification thousands of times over different crops.

Why "real-time" actually means something here Because there's only one network evaluation per image, YOLO's base model runs at 45 frames per second on a Titan X GPU, and a smaller "Fast YOLO" variant pushes past 150 FPS — under 25 milliseconds of latency per frame. That's fast enough to process a live video stream, not just a folder of photos.
03 · Splitting The Image Into Responsibility Zones

The S×S grid, and who's responsible for what

YOLO divides the input image into an S×S grid. The rule that makes everything else work: if the center of an object falls into a grid cell, that cell — and only that cell — is responsible for detecting it.

For PASCAL VOC, the paper uses \(S = 7\), giving a 7×7 grid over a 448×448 input image — so each cell covers a 64×64 pixel patch. Each grid cell predicts B bounding boxes (B = 2 in the original paper) plus a confidence score for each box, and one set of class probabilities per cell — shared across whichever boxes that cell predicts.

Why B = 2 matters at training time Each grid cell predicts two candidate boxes, but only one predictor is "responsible" per object — whichever of the two currently has the highest IOU with the ground truth. This forces specialization: over training, each predictor gets better at particular sizes or aspect ratios, which improves overall recall.
04 · The Coordinate Math, Worked By Hand

How a bounding box actually gets parameterized

This is the part that looks fussy on paper but is genuinely simple once you plug in real numbers — exactly the way my class notes walk through it.

Each bounding box is 5 numbers: \(x, y, w, h\), and confidence. The center coordinates \((x,y)\) are offsets relative to the grid cell that owns the box — not the whole image — so they're bounded between 0 and 1. Width and height, \(w\) and \(h\), are predicted relative to the whole image, also bounded 0–1.

Take a concrete example: a 448×448 image, \(S=7\) so each cell is 64×64 pixels, and an object whose bounding box runs from top-left corner \((192, 256)\) to bottom-right corner \((200, 311)\)... actually the box center is what falls at \((200, 311)\) inside the cell whose own top-left corner sits at \((x_a, y_a)\). The offset math:

\[ \Delta x = \dfrac{x - x_a}{64} \qquad \Delta y = \dfrac{y - y_a}{64} \qquad \Delta w = \dfrac{w}{448} \qquad \Delta h = \dfrac{h}{448} \]
QuantityValue
Δx = (200 − 192) / 64≈ 0.13
Δy = (311 − 256) / 64≈ 0.87

Every prediction is normalized this way for the same reason positional encoding gets normalized elsewhere: it keeps every number in a consistent, learnable range no matter where in the image or how large the object is.

05 · Everything The Network Outputs, In One Tensor

The 7×7×30 prediction tensor

Here's where all the pieces above get packed into one single output the network produces in a single forward pass.

Per grid cell, the model predicts: 2 bounding boxes × 5 values each (\(x, y, w, h,\) confidence), plus \(C = 20\) class probabilities shared across the cell. That's \(2 \times 5 + 20 = 30\) numbers per cell, and \(7 \times 7 = 49\) cells — giving a final output tensor of shape:

\[ S \times S \times (B \times 5 + C) = 7 \times 7 \times 30 \]
Index rangeContent
1st–5thBox #1 → x, y, w, h, confidence
6th–10thBox #2 → x, y, w, h, confidence
11th–30th20 conditional class probabilities, Pr(Classi | Object)

One subtlety worth sitting with: class probabilities are conditioned on an object being present — \(Pr(\text{Class}_i \mid \text{Object})\), not \(Pr(\text{Class}_i)\) outright. They only mean anything once you know something is actually there. If no object's center falls in a cell, every one of that cell's 30 values should be driven toward zero — which is exactly the "no object → all zeros" label encoding used during training.

06 · How Confident Is The Model, Really

Confidence score and class-specific confidence

Confidence isn't just "how sure am I an object is here" — it's defined precisely as a product of two things.

\[ \text{confidence} = Pr(\text{Object}) \times IOU^{\text{truth}}_{\text{pred}} \]

If no object exists in that cell, confidence should be exactly zero. Otherwise, it should equal the IOU (intersection-over-union) between the predicted box and the actual ground-truth box — so confidence reflects both "is something here" and "how well does my box actually fit it."

At test time, this confidence gets multiplied against the cell's conditional class probability to produce one final, class-specific score per box:

\[ Pr(\text{Class}_i \mid \text{Object}) \times Pr(\text{Object}) \times IOU^{\text{truth}}_{\text{pred}} = Pr(\text{Class}_i) \times IOU^{\text{truth}}_{\text{pred}} \]

That single number encodes everything you need to threshold on: how likely this box contains that specific class, and how tightly the box fits.

07 · The Network Itself

24 conv layers, 2 FC layers, one tensor out

YOLOv1's architecture is inspired by GoogLeNet, but simplified: instead of inception modules, it alternates 1×1 reduction layers with 3×3 convolutions to cut computation before the expensive layers.

StageLayersOutput
Stem7×7×64-s2 conv → maxpool112×112×64
Block 23×3×192 conv → maxpool56×56×192
Block 31×1/3×3 reduction pairs → maxpool28×28×512
Block 4 (×4)1×1/3×3 pairs → maxpool14×14×1024
Block 5 (×2 + stride-2)3×3×1024 convs7×7×1024
Headflatten → FC 4096 → FC 1470 → reshape7×7×30

Notice the last step: the final conv feature map is 7×7×1024 = 50,176 values. That gets flattened, passed through a fully connected layer down to 4096, then another FC layer up to 1470 values — and 1470 reshapes exactly into \(7 \times 7 \times 30\). Those two FC layers are the part that later gets torn out entirely in YOLOv2, for reasons we'll get to.

Leaky ReLU, and why "leaky" Every layer except the final one uses leaky rectified linear activation: \(\phi(x) = x\) if \(x>0\), else \(0.1x\). Regular ReLU zeroes out anything negative entirely; the small 0.1 slope keeps a trickle of gradient flowing even for negative activations, which helps avoid dead neurons during training.

Training happens in two stages: the first 20 convolutional layers are pretrained on ImageNet classification at 224×224 (reaching 88% top-5 accuracy — comparable to GoogLeNet). Then four new conv layers and two FC layers, randomly initialized, get added on top, and the input resolution doubles to 448×448 for the actual detection training — because detection needs the fine-grained visual detail that a smaller image throws away.

Fast YOLO is the same recipe with a smaller network: 9 convolutional layers instead of 24, and fewer filters per layer. Every training and testing parameter otherwise stays identical.

08 · How Training Actually Grades A Prediction

The multi-part loss function, piece by piece

YOLO optimizes a single sum-squared error loss — but it's really four different losses stitched together, each weighted differently on purpose.

Two problems with plain sum-squared error had to be fixed. First: in any image, most grid cells contain no object at all, so their confidence targets are zero — and that overwhelming majority of "no object" gradient can drown out the signal from the few cells that actually matter. Second: sum-squared error treats a small coordinate error the same whether the box is tiny or huge, but a small deviation matters far more, proportionally, for a small box than a large one.

Fix 1 — reweight object vs. no-object cells

Two constants control this: \(\lambda_{coord} = 5\) (boost the weight on box-coordinate errors) and \(\lambda_{noobj} = 0.5\) (shrink the weight on confidence errors from empty cells).

Fix 2 — predict square roots of width and height

Instead of penalizing \(w\) and \(h\) directly, the loss penalizes \(\sqrt{w}\) and \(\sqrt{h}\). A fixed absolute error in the square-root space corresponds to a smaller relative error for large boxes and a larger relative error for small boxes — which is exactly the behavior we want, since a few pixels off matters much more for a small object than a large one.

\[ L^{box}_{i,obj} = (\Delta x^* - \Delta \hat{x})^2 + (\Delta y^* - \Delta \hat{y})^2 + \left(\sqrt{\Delta w^*} - \sqrt{\Delta \hat{w}}\right)^2 + \left(\sqrt{\Delta h^*} - \sqrt{\Delta \hat{h}}\right)^2 \]

Alongside the box loss sit a confidence loss and a classification loss, each contributing their own term, only for cells that actually contain an object:

\[ L^{conf}_{i,obj} = (c^*_i - \hat{c}_i)^2 \qquad L^{cls}_{i,obj} = \sum_{c=1}^{20} (p_{i,c} - \hat{p}_{i,c})^2 \]

Worked with real numbers from a single object cell: say the ground-truth confidence is 1.0 and the network currently predicts 0.9 — that's a confidence loss of \((0.9-1.0)^2 = 0.01\). For the 20-way class vector, if the true class is "person" (index 14, target 1.0) and everything else is 0, but the network currently outputs 0.1 for one wrong class and 0.8 for "person": the classification loss sums \((0.1-0.0)^2 + \cdots + (0.8-1.0)^2 + \cdots\) across all 20 slots.

And for the cells that have no object — the vast majority — only a confidence loss applies, scaled down by \(\lambda_{noobj}\). If a no-object cell's two box predictors currently output confidences of 0.5 and 0.2 (should both be 0):

\[ L_{i,noobj} = (0.5-0.0)^2 + (0.2-0.0)^2 \]

Put together, the full loss sums over every grid cell and every box predictor:

\[ L = \lambda_{coord}\sum_{i=0}^{S^2}\sum_{j=0}^{B} \mathbb{1}^{obj}_{ij}\Big[(x_i-\hat{x}_i)^2+(y_i-\hat{y}_i)^2\Big] + \lambda_{coord}\sum_{i=0}^{S^2}\sum_{j=0}^{B}\mathbb{1}^{obj}_{ij}\Big[(\sqrt{w_i}-\sqrt{\hat{w}_i})^2+(\sqrt{h_i}-\sqrt{\hat{h}_i})^2\Big] \] \[ + \sum_{i=0}^{S^2}\sum_{j=0}^{B}\mathbb{1}^{obj}_{ij}(C_i-\hat{C}_i)^2 + \lambda_{noobj}\sum_{i=0}^{S^2}\sum_{j=0}^{B}\mathbb{1}^{noobj}_{ij}(C_i-\hat{C}_i)^2 + \sum_{i=0}^{S^2}\mathbb{1}^{obj}_i\sum_{c\in classes}(p_i(c)-\hat{p}_i(c))^2 \]

where \(\mathbb{1}^{obj}_i\) marks whether an object's center falls in cell \(i\), and \(\mathbb{1}^{obj}_{ij}\) marks which of the \(j\) box predictors in that cell is "responsible" — the one with the highest current IOU against the ground truth.

Training schedule, briefly 135 epochs on VOC 2007+2012, batch size 64, momentum 0.9, weight decay 0.0005. The learning rate is raised slowly from \(10^{-3}\) to \(10^{-2}\) over the first epochs — starting high causes divergence from unstable gradients — then held at \(10^{-2}\) for 75 epochs, \(10^{-3}\) for 30, and \(10^{-4}\) for the final 30. Dropout (rate 0.5) after the first FC layer, plus random scaling, translation, and HSV exposure/saturation jitter, prevent overfitting.
09 · Cleaning Up Duplicate Detections

IOU and Non-Max Suppression

On PASCAL VOC, YOLO predicts 98 total bounding boxes per image (7×7 cells × 2 boxes). Most of the interesting cleanup happens after that, not during it.

Intersection over Union (IOU) IOU measures how much two boxes overlap: the area where they intersect, divided by the total area they cover together. IOU = 0 means no overlap at all; IOU = 1 means the two boxes are identical. An IOU above 0.5 against ground truth is generally treated as a "good" prediction.

Because the grid enforces spatial diversity, an object is often cleanly claimed by exactly one cell. But large objects, or ones straddling a cell boundary, can get detected by more than one cell — producing several overlapping boxes for the same object. Non-Max Suppression (NMS) cleans this up: for each cluster of overlapping boxes above an IOU threshold, keep only the one with the highest confidence score and discard the rest.

NMS isn't as critical for YOLO's performance as it is for R-CNN or DPM — the grid design already limits duplicate detections — but it still adds a real 2–3% mAP improvement when applied.

10 · Was It Actually Good?

mAP, FPS, and how YOLOv1 actually performed

Two metrics matter here, and they're always in tension: mAP measures accuracy, FPS measures speed. Faster models tend to be less accurate; more accurate models tend to be slower. YOLO's whole philosophy is finding the best point on that curve.

mAP — mean Average Precision Precision: of all predicted boxes, what fraction were actually correct? Recall: of all real objects, what fraction did the model find? Average Precision combines both into one number per class; mAP averages that across all 20 classes. Max possible = 100.

That generalization gap matters enormously for real deployment, where the world never looks exactly like the training set.

11 · Where It Actually Broke

YOLOv1's three real limitations

These three weaknesses matter more than any other part of the paper — because every one of them became the direct research agenda for v2 and v3.

1. A hard ceiling of 49 objects

A 7×7 grid has 49 cells, and each cell can claim at most one object. Fine for a typical photo; immediately limiting for a crowded street, a flock of birds, or a packed stadium.

2. Small objects in groups get missed

Directly related to the ceiling above — if several small objects cluster together, their centers can fall in the same cell, and only one gets detected. Worse, by the final 7×7 feature map, each cell already represents a 64×64 pixel region of the original image — too coarse to resolve small objects that fit entirely within it.

3. Poor localization

Predicted boxes were often noticeably off from the true object boundary — for two compounding reasons: the fully-connected layers at the end flatten everything and lose precise spatial information, and the two box predictors per cell had no predefined shape to start from, so the network had to learn box shapes completely from scratch, a much harder optimization problem.

YOLO made more than twice the localization mistakes of Fast R-CNN — but Fast R-CNN was almost 3× more likely to hallucinate a detection on pure background. Neither model is strictly better; they fail in opposite ways, which is exactly why combining the two (rescoring Fast R-CNN detections using YOLO to kill background false positives) boosted the combined mAP to 75.0%, a 3.2-point gain over Fast R-CNN alone.

12 · One Year Later, Every Weakness Addressed

YOLOv2 (YOLO9000)

Released in 2016, YOLOv2's guiding question was simple: can we make YOLO both more accurate and still fast? Every change below is a direct answer to something on the limitations list.

Batch normalization

Added after every conv layer to keep activations in a stable, consistent range. Result: +2% mAP, and dropout could be removed entirely — the network became more stable and trained faster.

High-resolution classifier

YOLOv1 pretrained at 224×224 then jumped straight to 448×448 detection training — forcing the network to adapt to a new resolution and learn detection simultaneously. YOLOv2 first fine-tunes the backbone on ImageNet at 448×448 for 10 epochs before detection training even starts. Result: +4% mAP, for a genuinely simple change.

Fully convolutional — no more FC layers

The FC layers are removed entirely. Predictions come directly from the final conv layer. This preserves spatial information all the way to the output, lets the network accept any input size instead of being locked to 448×448, and cuts parameters — making the model faster too.

Anchor boxes, done properly this time

Rather than learning box shapes from nothing, YOLOv2 runs k-means clustering on the widths and heights of every ground-truth box in the training set, producing the shapes that actually occur in the data — tall and thin for people, wide and short for cars, small and square for faces. 5 anchor boxes per cell (up from v1's 2, which weren't properly defined at all). The network now only needs to learn small adjustments to a realistic starting shape, not an entire shape from scratch.

New backbone: Darknet-19

A cleaner 19-layer backbone purpose-built for the speed/accuracy tradeoff, using aggressive 1×1 convolutions to cut computation before the expensive 3×3 layers.

Multi-scale training

Since the network is now fully convolutional, it can accept different input sizes. Every 10 batches during training, the input resolution is randomly resampled — from 320×320 up to 608×608. At inference, you can then choose a small input for speed (mobile/edge) or a large input for accuracy — same trained weights, different tradeoff, just by changing the input size.

Passthrough layer

To help with small objects, a passthrough layer takes the earlier, higher-resolution 26×26 feature map and concatenates it with the final 13×13 map — giving the detection head both a zoomed-out view (good for large objects) and a zoomed-in view (good for small ones) at once. This is the direct precursor to full multi-scale detection in v3.

The "9000" — combining detection and classification data Detection datasets like Pascal VOC only cover 20 classes; COCO covers 80. The authors built a hierarchical WordTree that organizes classes from both detection and classification datasets (like ImageNet) into one tree, letting the model train on both simultaneously. The result: YOLO9000 could detect over 9000 categories — hence the name.

13 · Solving Multi-Scale Detection For Real

YOLOv3

2018. If v2 fixed training and anchors, v3 tackles the hardest problem still standing: detecting both tiny and large objects reliably in the same image.

YOLOv2's passthrough layer was a partial patch. The deeper issue remained: a single feature map cannot be simultaneously good at detecting small and large objects. Early layers have high spatial resolution — good for fine detail, good for small objects. Late layers have low resolution but rich semantic context — good for large objects. v2 only predicted from the final, low-resolution map.

YOLOv3's answer: predict from three different scales at once.

ScaleFeature map sizeBest for
Large13×13Large objects — cars, buses, near-field people
Medium26×26Medium objects
Small52×52Small objects — faces, birds, distant pedestrians

Each scale gets its own 3 anchor boxes, so 3 anchors × 3 scales = 9 total anchors, still chosen via k-means but now split across scales.

The key trick is what "upsample and concatenate" actually combines: deep features from later layers carry rich semantic understanding ("this looks like a dog"), while shallow features from earlier layers carry precise spatial detail ("it's exactly here, and this big"). Merging them gives both at once — understanding and location — which is exactly what accurate detection needs. This structure is a Feature Pyramid Network (FPN), a technique YOLOv3 adopted from the broader research community rather than inventing from scratch.

New backbone: Darknet-53

Darknet-19 is replaced with a much deeper 53-layer backbone, packed with residual (skip-connection) blocks — the same idea that lets ResNets train cleanly at huge depth without gradients vanishing.

BackboneTop-1 accuracyFPS
ResNet-10177.1%53
ResNet-15277.6%37
Darknet-5377.2%78

Matches ResNet-152's accuracy at roughly twice the speed — the whole point of designing a backbone specifically for this tradeoff rather than reusing one built for classification benchmarks alone.

Softmax → independent sigmoid for class prediction

Softmax forces class probabilities to sum to 1 — assuming an object belongs to exactly one class, mutually exclusive with all others. Real datasets, especially with hierarchical labels, break that assumption ("dog" and "animal" can both be true). YOLOv3 uses an independent sigmoid per class instead, letting each be true or false on its own — which also makes multi-label detection possible.

Doing the output-shape math

Each box vector is now \(5 + 80 = 85\) numbers (x, y, w, h, confidence, plus 80 COCO classes):

The mAP jump from 21.6 to 33.0 on COCO is driven almost entirely by small objects finally being detected well for the first time — the exact problem the whole multi-scale redesign targeted.

Why v3 became the long-running default Fast enough for real-time on a decent GPU, accurate enough for most real applications, well documented and open source, and — for the first time — genuinely good at both tiny and large objects in the same frame. That combination is why most detection tutorials and production systems from 2018–2021 built on YOLOv3, and why it's still referenced constantly today.

If you forget everything else, remember this

  • One pass, not hundreds. The whole point of YOLO is reframing detection as regression, not repeated classification.
  • The grid rule: whichever cell contains an object's center owns detecting it.
  • Confidence = Pr(Object) × IOU — both "is it there" and "does the box fit" in one number.
  • v1's three scars — object cap, small-object clusters, poor localization — map directly onto v2 and v3's fixes.
  • v2 = better training tricks + real anchors + fully convolutional.
  • v3 = multi-scale detection via a feature pyramid, finally solving small objects.
  • NMS and IOU thresholding turn hundreds (or thousands) of raw candidate boxes into a handful of final detections.
