MY NOTES ON HOW NEURAL NETS LEARN, WRITTEN OUT PROPERLY

Nobody tells the network what an edge looks like. It figures that out by getting yelled at, math-style.

I went through the 3Blue1Brown neural network series a few times, paused on a lot of frames, and wrote my own notes alongside it. This is that, cleaned up: what a neuron actually is, how a network makes a guess, and — the part that actually took me a few rewatches to get — how it works backward from "you were wrong" to "here's exactly which of the 13,000 numbers to nudge, and by how much."

01 · Starting Small

First, what even is a neuron?

Strip away the brain metaphor and a neuron is almost boringly simple: it's a box that holds one number, somewhere between 0 and 1.

That's basically it. People hear "neuron" and picture something biological and mysterious, but in a network it's just a slot for a number. For the very first layer of neurons, that number is just the brightness of one pixel — 0 for pitch black, 1 for white, decimals in between for gray. That's the network's entire "vision."

A small caveat, so I'm not lying to you "Between 0 and 1" is only strictly true when the neuron's activation function is sigmoid, which squashes everything into that range on purpose. Later on we'll meet ReLU, which is the more common choice today — and ReLU happily outputs any number 0 or above, with no upper ceiling. So the more accurate version of "what is a neuron" is: a box holding one number, where that number is non-negative, and how tightly it's capped depends on which activation function the layer is using. For the input layer specifically (raw pixel brightness), 0–1 is still exactly right.

Here's the example everyone uses, and for good reason: take a 28×28 pixel image of a handwritten digit. That's 784 pixels, so the input layer has 784 neurons. The network's job is to look at those 784 brightness values and spit out a guess: which digit is this?

Activation, in plain words Whatever number is currently sitting inside a neuron. High activation (close to 1) basically means "this neuron is lit up." Low activation (close to 0) means it's quiet. That's the entire vocabulary you need for now.

Between the input and the final answer, there are hidden layers — in the classic version of this example, two layers of 16 neurons each. Why 16? In this specific toy example, mostly because it fit nicely on a screen — but don't read that as "the number doesn't matter." In real projects, hidden layer size is a genuine hyperparameter, something you choose deliberately and often tune by experiment: too few neurons and the network doesn't have enough room to represent the patterns in your data (underfitting), too many and it has so much spare capacity that it starts memorizing quirks of the training set instead of learning the general pattern, on top of just being slower to train. So "16" was arbitrary here, but "how many neurons should this layer have" is a real, considered decision in practice. What matters conceptually is the idea underneath it: activations in one layer get combined to produce activations in the next, all the way to an output layer of 10 neurons — one per digit, 0 through 9. Whichever of those 10 ends up brightest is the network's answer.

02 · The Hope, Before The Reality

Why bother with layers at all?

Because that's roughly how you and I actually read handwriting — by noticing small pieces and stacking them up into a bigger idea.

Think about how you tell an 8 from a 9. You're not comparing pixel-by-pixel against some mental template. You're noticing: there's a loop up top, and another loop below it — that's an 8. One loop and a line trailing off to the side — probably a 9. You're unconsciously breaking the digit into smaller, reusable pieces: loops, lines, curves.

The hope behind a layered network is that it might learn to do something similar, automatically:

  • First hidden layer picks up tiny edges — a bright pixel next to dark ones.
  • Second hidden layer combines edges into shapes: loops, long strokes, corners.
  • Output layer combines those shapes into a final decision.

And this isn't just an image-recognition trick. Anything that breaks into layers of abstraction works this way — speech recognition turns raw sound into phonemes, phonemes into words, words into meaning. Same shape, different domain.

03 · The Actual Mechanics

Okay, but how does one layer actually produce the next?

Two ingredients do all the work here: weights and biases. Everything fancy that a network does eventually traces back to these two numbers.

A weight sits on a connection between two neurons and says how much that connection matters. A big positive weight means "when you're bright, make me brighter too." A negative weight means the opposite — "when you're bright, that's evidence I should be dimmer." A bias sits on the receiving neuron and shifts how easily it gets excited — think of it as the neuron's personal threshold for "okay, NOW I'll light up."

Mechanically, a neuron takes every incoming activation, multiplies each by its weight, adds them all up, then adds the bias. That raw number is usually called \(z\):

\[ z = (w_1 x_1 + w_2 x_2 + \cdots + w_n x_n) + b \]

\(z\) can be anything — it has no respect for the 0-to-1 rule activations are supposed to follow. So before it's allowed to become the neuron's actual activation, it gets squeezed through a function that forces it into a sane range. More on that function in a second. The whole pipeline, start to finish, looks like this:

input → multiply by weight → add bias → squash with activation function → output

A concrete example: hand-building an edge detector

Say you wanted one specific neuron to fire only when there's an edge in one particular patch of the image — nothing fancy, just "is there a bright strip surrounded by dark pixels here?" You could actually hand-craft this with weights alone: set almost every weight to roughly zero, give small positive weights to the pixels inside that patch, and small negative weights to the ring of pixels surrounding it.

And the bias? If you only want this neuron to fire once that sum gets past, say, 10 — not just any positive number — you set the bias to −10. So, in short: weights decide what pattern a neuron is searching for, and bias decides how convinced it needs to be before it actually reacts.

Why this gets out of hand fast

Every neuron in a layer connects to every neuron in the previous layer, and each one of those connections gets its own weight. Just for the jump from the 784-pixel input layer to the first 16-neuron hidden layer, you're already looking at:

\(784 \times 16 = 12{,}544\) weights, plus 16 biases — and that's only one layer transition.

Add up every transition in this small example network and you land at around 13,000 weights and biases total. "Training the network" is really just a fancy way of saying: find good values for those 13,000 numbers. Nobody sets them by hand — that would be insane. They start out random, and the network slowly tunes them through the process we're building up to.

04 · Squashing The Number Into Something Useful

Activation functions

The raw weighted sum \(z\) is unruly — it could be 500, it could be -8000. The activation function's job is to tame it, and along the way, give the network the ability to learn things a straight line never could.

Quick example with ReLU, since it's the easiest to picture: feed it −7, you get 0. Feed it 12, you get 12 back, untouched. That's the entire function. No exponentials, no curves — which is exactly why it's cheap to compute and why it's become the default choice for hidden layers in most modern networks.

Why even bother with a nonlinear function?

This part trips people up, so let me try to make it intuitive. If every layer were just a weighted sum with nothing nonlinear afterward, then stacking ten layers would mathematically collapse into the same thing as one layer. You'd gain nothing from depth — it'd just be one big linear equation wearing a costume. The nonlinearity is what actually lets the network bend its decision-making into curvy, complicated shapes instead of one flat plane.

So why did everyone abandon sigmoid for ReLU?

Short version, stated plainly: sigmoid was the original activation function; modern networks almost universally use ReLU for hidden layers instead, because it trains faster and sidesteps a specific failure mode called the vanishing gradient problem. Here's the longer version, since we've already done the calculus to actually understand it.

Sigmoid was the original choice, partly because it felt biologically poetic, a neuron that's "off," "on," or somewhere gently in between. The problem shows up once a network gets deep, and it's a direct consequence of backpropagation's chain rule, which you'll see worked out in full a bit further down. Every time a gradient signal passes backward through one more layer, it gets multiplied by that layer's activation function derivative, \(\sigma'(z)\). For sigmoid, that derivative tops out at exactly 0.25, its absolute best case, and it's far smaller than that almost everywhere else, since sigmoid goes nearly flat for big positive or negative inputs.

Vanishing gradient problem Chain a handful of these multiplications together (one per layer, on the way backward through the network) and you're multiplying several numbers that are all 0.25 or smaller. Even with a generous, optimistic 0.25 at every layer, ten layers deep means the gradient reaching the earliest layers has been shrunk by roughly \(0.25^{10} \approx 0.00000095\), essentially nothing. Those early layers stop receiving any meaningful "here's how to improve" signal at all, and effectively stop learning, even while later layers keep training normally.

ReLU sidesteps this almost by accident. Its derivative is either exactly 0 (for negative inputs) or a clean, constant 1 (for positive ones). No shrinking multiplication, no flattening curve: a gradient passing through an active ReLU neuron comes out the other side exactly as strong as it went in. That one practical difference is a big part of why deep networks, ones with dozens or hundreds of layers, became trainable at all. It's also exactly why the cheat sheet at the end of this post says "hidden layers → ReLU" without much hesitation.

05 · Tidying Up The Notation

Vectors, matrices, and "forward propagation"

Writing every single neuron's equation out by hand would be miserable. Linear algebra exists to let us say the same thing far more compactly.

Stack a layer's activations into a column vector \(\mathbf{a}\). Stack every weight feeding into the next layer into a matrix \(W\) (each row holds one neuron's incoming weights). Stack the biases into a vector \(\mathbf{b}\). Then the jump from one layer to the next, all 16 or all 784 neurons at once, is just:

\[ \mathbf{a}^{(L)} = \sigma\big(W^{(L)} \mathbf{a}^{(L-1)} + \mathbf{b}^{(L)}\big) \]

where \(\sigma\) gets applied to every entry separately. It looks denser, but it's the exact same arithmetic as before — just written so a computer (and your own brain) can treat "advance one layer" as a single operation instead of hundreds of tiny ones.

Running an input through this, layer after layer, until you fall out the other end with a guess, is called forward propagation. That's really all the phrase means: push the data forward, get a prediction.

06 · Putting A Number On "Wrong"

The loss (cost) function

A brand-new network has random weights and biases, so naturally its first guesses are garbage. Before you can fix that, you need a precise way to measure exactly how garbage.

Feed in a picture of a 5. You want the output layer to show activation 1 on the "5" neuron and 0 everywhere else. In reality, especially early on, you'll get some chaotic mess — maybe 0.3 here, 0.6 there, nothing close to the clean answer you wanted. One straightforward way to score that gap, for a single example, adds up the squared distance between each output \(a_j\) and its target \(y_j\):

\[ C_0 = \sum_{j=0}^{9} (a_j - y_j)^2 \]

Confident and correct → small \(C_0\). Confident and wrong, or just hopelessly unsure → big \(C_0\). Average that across every training example you've got — tens of thousands of them, in the standard MNIST digit dataset — and you get the full cost function:

\[ C(W, \mathbf{b}) = \frac{1}{n}\sum_{i=1}^{n} C_0^{(i)} \]

Look closely at what this function takes as input. Not an image — the ~13,000 weights and biases. It's a function sitting one level above the network itself, and the entire goal of training boils down to one sentence: find the weights and biases that make \(C\) as small as it can possibly be. Everything from here on is just "how do we actually do that."

Quick honesty check before we go further The squared-error version above (often called MSE, mean squared error) is the one we'll actually carry through the calculus in the backpropagation section, because it keeps the math clean and easy to follow step by step. In real, modern classification networks, though, you'll almost always see cross-entropy used instead — it's covered just below. The two play the same role (measuring "how wrong") and the backpropagation logic works the same way for both; only the exact derivative formulas at the very last step change. I'm using squared error here purely as a teaching tool, the same way 3Blue1Brown does in the original videos.

One small but useful side-note: squared error works fine, but in practice people often reach for cross-entropy instead for classification problems, especially once softmax is involved:

\[ C_0 = -\sum_{j} y_j \log(a_j) \]

It punishes a confident wrong answer harder than squared error does, which in practice tends to speed learning up. Rule of thumb: sigmoid + binary classification pairs with binary cross-entropy; softmax + multi-class pairs with categorical cross-entropy.

07 · The Strategy For Getting Better

Gradient descent — basically, walking downhill

A function of 13,000 variables is impossible to picture in your head. Lucky for us, the strategy for minimizing it is identical to the one you'd use for a function of a single variable.

Picture the cost function as a landscape, hills and valleys, and a ball sitting somewhere on it. You want that ball to roll down into a valley — a spot where the cost is low. At any point you're standing, the slope tells you which way is downhill: positive slope, step left; negative slope, step right. Keep checking the slope and taking small steps, and you'll eventually settle into a valley.

For functions of more than one variable, "slope" gets upgraded to the gradient, written \(\nabla C\) — a vector pointing in the direction the function increases fastest. Flip its sign, \(-\nabla C\), and now it points downhill. Its length also tells you how steep that downhill direction is.

None of that logic changes when you scale up from 2 dimensions to 13,000. Line up every weight and bias into one enormous vector \(\theta\), and \(-\nabla C(\theta)\) is a single direction in that 13,000-dimensional space telling you precisely how to tweak every one of those numbers to drop the cost as fast as possible. With a step size \(\eta\) (the learning rate), one update looks like:

\[ \theta \leftarrow \theta - \eta \, \nabla C(\theta) \]

Repeat that update again and again — that whole loop is gradient descent. And here's a detail worth sitting with: the size of each individual component inside \(\nabla C\) isn't just a "go up or down" signal. Its magnitude tells you how much that one weight actually matters for the current cost — which of the 13,000 dials is worth turning, and which one barely makes a difference. We'll lean on that idea heavily in the next section.

Side note This is also why artificial neurons use smooth activations instead of a strict on/off switch like real neurons. Gradient descent needs an actual slope to follow. A jagged, binary surface doesn't really have one.
08 · The Part That Actually Took Me A Few Tries

Backpropagation — figuring out who's to blame

Gradient descent tells you the plan: walk downhill. It doesn't tell you which way downhill is in a 13,000-dimensional space. That's backpropagation's entire job — and once you walk through it slowly, it's not actually that scary.

Start with what one training image "wants"

Say you feed in an image of a handwritten 2. The network hasn't been trained yet, so the output layer is a mess — random-ish activations like 0.5, 0.8, 0.2, scattered across all ten output neurons, instead of a clean 1 on "2" and 0 everywhere else.

We can't reach into the network and directly set that "2" neuron's activation to 1. We only get to adjust weights and biases. But it's useful to write down what this neuron wishes would happen — and there are exactly three levers that could push its activation up:

  • Increase the bias on that neuron. The simplest lever — directly lowers its threshold to fire.
  • Increase the weights coming from whichever previous-layer neurons are already bright, since a weight's effect scales with how active the neuron behind it is. Bumping up a weight attached to a dim neuron barely moves anything; bumping up a weight attached to a bright one moves a lot.
  • Change the previous layer's activations — make everything connected with a positive weight brighter, and everything connected with a negative weight dimmer.

That third option is the interesting one, because we can't touch it directly either — but we can write down the wish, and pass it backward, like a memo: "hey, second-to-last layer, here's how I'd like you to change." Recurse that same logic one layer further back, and the layer after that, and you've got the whole algorithm in spirit.

Every output neuron has its own opinion

Here's the catch: there isn't just one output neuron with wishes. All ten of them do. The "2" neuron wants to get brighter. The "8" neuron, the "3" neuron, all the others, want to get dimmer — and each one has its own idea about how the second-to-last layer should change to make that happen, scaled by how strong the relevant weight is and by how far off that output currently is from its target.

So you don't just listen to one neuron's request. You add together every output neuron's wish for that hidden layer, and that combined signal is what actually propagates backward.

A small, honest tangent There's an old idea in neuroscience that fits this surprisingly well: cells that fire together end up wired together. The biggest weight increases happen between neurons that are already active and the ones we're trying to make more active. I'm not saying this proves anything about how real brains work — that's a much bigger claim than I can make — but the match with the math here is fun to notice.

If we only listened to one image, we'd have a problem

If we only ever optimized for that one picture of a 2, the network would happily learn to call everything a 2 — that's the fastest way to satisfy a single training example. So this entire backward pass — calculate what every weight and bias "should" change by — gets run separately for every training example, and then all those desired changes get averaged together.

Picture an enormous spreadsheet: one column per training image, one row per weight. Each cell holds how much that one image thinks that one weight should change.

So when you hear "the negative gradient of the cost function," this is really all it is under the hood: take every training example's wish-list for the weights, average it row by row, and you've got your direction to step.

\[ -\nabla C(w_1, w_2, \dots, w_{13{,}001}) = \begin{bmatrix}-0.08\\+0.12\\-0.06\\ \vdots \\+0.04\end{bmatrix} \]

Now let's actually do the calculus

Everything above is the intuition. Here's the part where we earn it with real math. The trick that makes it manageable: don't try to write the calculus for the whole tangled network at once. Strip it down to the smallest possible version first, one neuron per layer, get totally comfortable there, and only then let extra neurons back in. (One more thing before we dive in: we're using squared error for \(C_0\) here, not cross-entropy, purely because it keeps the derivatives short and friendly. Swap in cross-entropy later and the overall shape of the algorithm, the chain rule, the backward recursion, all of it, stays exactly the same. Only the very last derivative, \(\partial C_0/\partial a^{(L)}\), changes its formula.)

Step 1 — the smallest network you can draw

One neuron per layer, four layers, nothing else going on:

Now zoom in further — just the very last connection, between the second-to-last neuron and the last one. That's the only piece of the picture we need for the first round of calculus:

Let \(y\) be the desired output for this last neuron. The cost for one training example, restricted to just this one neuron, is:

\[ C_0 = \big(a^{(L)} - y\big)^2 \]

where \(a^{(L)}\) is determined the same way every activation is:

\[ z^{(L)} = w^{(L)} a^{(L-1)} + b^{(L)}, \qquad a^{(L)} = \sigma\big(z^{(L)}\big) \]

Step 2 — follow the dependency chain

\(w^{(L)}\), \(a^{(L-1)}\), and \(b^{(L)}\) all feed into \(z^{(L)}\). \(z^{(L)}\) feeds into \(a^{(L)}\). \(a^{(L)}\), together with \(y\), feeds into \(C_0\):

w(L), a(L−1), b(L)  →  z(L)  →  a(L)  →  C₀  (also depends on y)

A tiny nudge to \(w^{(L)}\) causes a tiny nudge to \(z^{(L)}\), which causes a tiny nudge to \(a^{(L)}\), which causes a tiny nudge to \(C_0\). What we actually want is the ratio of that very last nudge to the very first one — that's the derivative we're hunting for:

\[ \frac{\partial C_0}{\partial w^{(L)}} \]

Step 3 — the chain rule turns one hard question into three easy ones

Because the effect of \(w^{(L)}\) has to travel through \(z\) and then through \(a\) before it reaches \(C_0\), the chain rule lets us break this one long-distance question into three short, local ones:

\[ \frac{\partial C_0}{\partial w^{(L)}} = \frac{\partial z^{(L)}}{\partial w^{(L)}} \cdot \frac{\partial a^{(L)}}{\partial z^{(L)}} \cdot \frac{\partial C_0}{\partial a^{(L)}} \]

Each one of those three factors is something you can just read off the equations above:

  • \(\dfrac{\partial C_0}{\partial a^{(L)}} = 2\big(a^{(L)} - y\big)\) — bigger when the network is more wrong.
  • \(\dfrac{\partial a^{(L)}}{\partial z^{(L)}} = \sigma'\big(z^{(L)}\big)\) — the activation function's own slope at that point.
  • \(\dfrac{\partial z^{(L)}}{\partial w^{(L)}} = a^{(L-1)}\) — same idea as before: a weight's leverage scales with how bright the neuron behind it already is.

Step 4 — put it together

\[ \frac{\partial C_0}{\partial w^{(L)}} = a^{(L-1)} \, \sigma'\big(z^{(L)}\big) \, 2\big(a^{(L)} - y\big) \]

That's the derivative for one training example, with respect to one weight. The bias works out almost identically — swap \(\partial z/\partial w = a^{(L-1)}\) for \(\partial z/\partial b = 1\), since the bias isn't multiplied by anything:

\[ \frac{\partial C_0}{\partial b^{(L)}} = 1 \cdot \sigma'\big(z^{(L)}\big) \cdot 2\big(a^{(L)} - y\big) \]

And remember, all of this is just for one training example. The real gradient — the thing gradient descent actually steps along — averages it over every example you've got, and stacks every weight's and every bias's derivative into one big vector:

\[ \frac{\partial C}{\partial w^{(L)}} = \frac{1}{n}\sum_{k=0}^{n-1} \frac{\partial C_k}{\partial w^{(L)}} \qquad\qquad \nabla C = \begin{bmatrix} \partial C/\partial w^{(1)} \\ \partial C/\partial b^{(1)} \\ \vdots \\ \partial C/\partial w^{(L)} \\ \partial C/\partial b^{(L)} \end{bmatrix} \]

Step 5 — propagating backward, one layer further

This is where the "back" in backpropagation earns its name. The same chain rule also tells us how sensitive the cost is to the previous layer's activation, \(a^{(L-1)}\) — just shifted over by one layer:

\[ \frac{\partial C_0}{\partial a^{(L-1)}} = \frac{\partial z^{(L)}}{\partial a^{(L-1)}} \cdot \frac{\partial a^{(L)}}{\partial z^{(L)}} \cdot \frac{\partial C_0}{\partial a^{(L)}}, \qquad \frac{\partial z^{(L)}}{\partial a^{(L-1)}} = w^{(L)} \]

We still can't directly change \(a^{(L-1)}\) — but now that we know how sensitive the cost is to it, we can repeat this exact same process one more layer back, treating \(a^{(L-1)}\) the way we just treated \(a^{(L)}\). That's the recursion: keep walking backward, one layer at a time, reusing the same three-step chain rule every time.

Everything above was for one neuron per layer. Now let's let real layers — with many neurons each — back in, and see exactly what changes.

Step 6 — now give layers more than one neuron

I'll be honest, this was the part that confused me the most on a first watch, because suddenly there are indices everywhere. But conceptually, almost nothing new happens — you're just summing over more paths. Index neurons in layer \(L\!-\!1\) (the source layer) with \(k\), and neurons in layer \(L\) (the destination layer) with \(j\). The weight connecting source neuron \(k\) to destination neuron \(j\) is \(w^{(L)}_{jk}\) — read it as "destination, then source." It looks reversed at first, but it matches how the weight matrix was set up earlier, so it's worth getting used to.

The cost for one example now sums over every output neuron, not just one:

\[ C_0 = \sum_{j=0}^{n_L - 1} \big(a_j^{(L)} - y_j\big)^2, \qquad z_j^{(L)} = w_{j0}^{(L)} a_0^{(L-1)} + w_{j1}^{(L)} a_1^{(L-1)} + w_{j2}^{(L)} a_2^{(L-1)} + b_j^{(L)} \]

The derivative with respect to one specific weight, \(\partial C_0/\partial w_{jk}^{(L)}\), looks exactly like before — it only ever refers to one path through one specific \(j\), so nothing new there:

\[ \frac{\partial C_0}{\partial w_{jk}^{(L)}} = \frac{\partial z_j^{(L)}}{\partial w_{jk}^{(L)}} \cdot \frac{\partial a_j^{(L)}}{\partial z_j^{(L)}} \cdot \frac{\partial C_0}{\partial a_j^{(L)}} \]

But here's the new wrinkle: a single neuron in layer \(L\!-\!1\) doesn't just feed into one neuron in the next layer — it feeds into all of them. So its derivative with respect to an activation in the previous layer, \(a_k^{(L-1)}\), has to add up its influence through every path it took to get there:

\[ \frac{\partial C_0}{\partial a_k^{(L-1)}} = \sum_{j=0}^{n_L - 1} \frac{\partial z_j^{(L)}}{\partial a_k^{(L-1)}} \cdot \frac{\partial a_j^{(L)}}{\partial z_j^{(L)}} \cdot \frac{\partial C_0}{\partial a_j^{(L)}} \]
What's actually new here Nothing about the chain rule itself changed. The only new idea is the sum: a neuron in layer L−1 affects the cost once through each neuron in layer L it connects to, so you add up its effect through every one of those connections. Everything inside the sum is the exact same three-step chain rule as the single-neuron case — we're just doing it once per path and adding the results.

Put it all together, generalize the layer index from \(L\) to any layer \(l\), and you get the full, general backpropagation formula — the one that actually runs inside real training code:

\[ \frac{\partial C}{\partial w_{jk}^{(l)}} = a_k^{(l-1)} \, \sigma'\big(z_j^{(l)}\big) \, \frac{\partial C}{\partial a_j^{(l)}} \]

where that last piece, \(\partial C/\partial a_j^{(l)}\), is exactly \(2(a_j^{(L)} - y_j)\) if \(l\) happens to be the very last layer — or, if there are layers after it, the same summed expression from above, reaching forward into the next layer:

\[ \frac{\partial C}{\partial a_j^{(l)}} = \sum_{i=0}^{n_{l+1}-1} w_{ij}^{(l+1)} \, \sigma'\big(z_i^{(l+1)}\big) \, \frac{\partial C}{\partial a_i^{(l+1)}} \]

And that's really the whole algorithm. You compute it starting at the output layer, where the formula is simple, and walk backward — each layer's answer depends on the layer after it, which you've already computed. That dependency on "the layer after it" is exactly why you have to go in this order, output to input, and exactly why it's called backpropagation rather than just "propagation."

Why nobody actually averages over the whole dataset every step

In theory, one proper gradient-descent step needs this averaged over every training image — all sixty-thousand-something of them in MNIST. In practice, that's brutally slow to do before every single update. So instead, people shuffle the data, chop it into small mini-batches (say, 100 images at a time), and compute a step using just one mini-batch.

It's not the true gradient, just a rough guess at it. Think of someone checking the slope under their feet every few steps and adjusting, instead of stopping to carefully survey the whole hill before each move. The guesses are noisier, but on average they still head downhill, and you save a ton of time. This shortcut has a name: stochastic gradient descent (SGD), and it's what almost every real network actually uses to train.

09 · Zooming Back Out

How all of this fits into one repeating loop

Forward propagation, the cost function, backpropagation, gradient descent — none of these are separate tools you reach for one at a time. They're one loop, run over and over, thousands of times.

StageQuestion it's answering
Forward propagationWhat did I predict?
Loss functionHow wrong was I?
BackpropagationWhy was I wrong — which weights caused it, and how much?
Gradient descentGiven that, how should I change the weights?

A few words you'll keep bumping into

Epoch One full pass through the entire training set.
Batch size How many training examples get processed before the weights actually update once.
Iteration One mini-batch processed, one update made. Dataset of 50,000 with batch size 500 gives you \(50{,}000 / 500 = 100\) iterations per epoch.

Quick worked example, since numbers stick better than definitions: dataset of 10,000 images, batch size 200, training for 10 epochs.

\[ \text{iterations per epoch} = \frac{10{,}000}{200} = 50 \qquad\qquad \text{total iterations} = 50 \times 10 = 500 \]
10 · One Number, Big Consequences

Learning rate — how big a step downhill

Remember \(\eta\) from the update rule, \(\theta \leftarrow \theta - \eta \nabla C(\theta)\)? That single number controls exactly one thing: step size. Get it wrong and training breaks in two very different ways depending on which direction you got it wrong.

One nice bit of self-correction baked into this: if your step size scales with the slope, then as you approach the bottom of a valley and the slope flattens out, your steps naturally shrink too. That's part of why gradient descent tends to settle gently into a minimum instead of flying past it — assuming your learning rate wasn't set wildly too high to begin with.

11 · Knowing When To Stop (Or Keep Going)

Overfitting and underfitting

A network chasing the lowest possible training loss can "win" in a hollow way — by memorizing the training set instead of actually learning anything that generalizes. There's an opposite failure too, and it's just as common for beginners to hit.

Overfitting Training accuracy looks great — 99%, say — but validation accuracy lags behind at 85%. The network has memorized the training images, noise and all, instead of learning the underlying pattern. It aced the practice test by memorizing the answer key, not by understanding the material.
Underfitting The opposite problem: training accuracy is mediocre (60%) and validation accuracy is just as mediocre (58%). The model never really learned the pattern in the first place — usually because it's too simple, undertrained, or the learning rate is off.

This is exactly why people split data three ways instead of two: train → validate → test. Training data adjusts the weights. The validation set just watches from the sidelines during training, flagging the moment things start overfitting. The test set never gets touched until the very end — it's your one honest, unbiased check on how the model really performs.

Early stopping Just stop training the moment validation performance stops improving, even if training loss is still happily dropping. Simple, and it works.
Dropout During training, randomly switch off a chunk of neurons on each pass — sounds destructive, but it stops the network from leaning too hard on any one neuron, forcing it to spread its understanding around. That redundancy is what helps it generalize.

The pattern that tells you things are going well

You don't need a fancy diagnostic for this — just watch four numbers move in the direction you want, at the same time: loss going down, accuracy going up, and the validation versions of both doing the same thing alongside them. If training metrics improve while validation metrics stall or worsen, that's your overfitting signal, every time.

12 · A Reality Check Worth Sitting With

Does it actually learn "edges, then loops, then digits"?

Remember the tidy hierarchy from earlier — pixels into edges, edges into loops, loops into digits? Here's the slightly humbling truth.

A plain network like the one we've described, two hidden layers of 16 neurons, trained this way on MNIST, does actually reach somewhere around 96–98% accuracy on digits it's never seen. That's a solid result for something this simple. But if you actually go and visualize the weights it learned — render them as a pixel grid the way we did back in FIG 2 — they mostly don't look like clean edge detectors. They look close to random, with only faint, loose structure if you squint.

Push it further: feed the trained network pure random static, no digit in sight. A truly "smart" system might respond with healthy uncertainty — low, evenly-spread activations across all ten outputs, basically shrugging. Instead, it'll often confidently insist the static is some specific digit, just as sure of that nonsense answer as it is of a real, clean 5. It found a valley in that enormous cost landscape that happens to classify real digits well — but it never developed anything like a real concept of "what a digit looks like." It only ever learned to tell the difference between examples inside a small, clean, centered world, because that's the only world its cost function ever showed it.

That's not a flaw to be embarrassed about. It's exactly why this simple network is treated as a foundation rather than a finished product. Every idea in this post (weighted sums, activation functions, cost functions, gradients, backpropagation) is the same machinery that more sophisticated modern architectures build on top of, ones that do manage to learn richer, more reusable features. You need this version first.

13 · A Quick Vocabulary Check

ANN vs. DNN — is this even the same thing?

Short answer: yes, mostly. It's less a hard boundary and more a matter of how many hidden layers you've stacked up.

ANNDNN
Hidden layersOne (sometimes the term gets used loosely for "a couple")Two or more, often many more
CapabilitySimpler — can still learn plentyMore powerful, can model deeper hierarchies of features
Compute costLowerHigher — more layers means more backward passes to compute

Everything explained in this post — weights, biases, forward propagation, backpropagation — applies identically whether you've got one hidden layer or fifty. "Deep learning" really just means "neural networks, but with enough hidden layers that it became worth giving the deeper version its own name."

14 · From Theory To Actually Running Code

What this looks like outside of a whiteboard

Everything above is the math. If you actually sat down to build this in code, the steps in practice look less like equations and more like a checklist.

  1. Load the dataset — for this example, that's MNIST: tens of thousands of labeled handwritten digit images.
  2. Visualize a few images — just to sanity-check that the data actually looks like what you expect before you trust anything else.
  3. Check the shape — confirm you really do have 28×28 images, and the right number of them.
  4. Normalize the pixels — rescale grayscale values from 0–255 down to 0–1, since that's the range the network expects.
  5. Flatten the images — unroll each 28×28 grid into a single list of 784 numbers, so it can feed into the input layer.
  6. One-hot encode the labels — turn a label like "4" into a vector that's 1 in the 4th slot and 0 everywhere else, matching the output layer's shape.
  7. Build the network — define the layers: input size, hidden layer sizes, output size, and which activation function each layer uses.
  8. Compile it — pick a loss function, an optimizer (a specific flavor of gradient descent), and which metrics you want tracked.
  9. Train it — run the forward-propagation → loss → backpropagation → update loop, epoch after epoch.
  10. Evaluate it — check accuracy on data it's never seen, the test set, not the training set.
  11. Build a confusion matrix — see exactly which digits get mistaken for which other digits.
  12. Look at what it got wrong — pull out the misclassified images by hand and actually look at them. More often than not, you'll find a few that would confuse a human too.

None of these steps require new theory — they're just the practical scaffolding around the actual learning algorithm, the part that turns "I understand backpropagation" into "I have a working model."

If you forget everything else, remember this

  • Forward propagation makes a guess.
  • The loss function scores how bad that guess was.
  • Backpropagation works backward through the chain rule to figure out exactly how much each of the ~13,000 weights and biases is to blame.
  • Gradient descent nudges every weight a little, in whichever direction actually helps.
  • Do that thousands of times, in small mini-batches, while a validation set quietly watches to make sure you're learning the pattern and not just memorizing the homework.
