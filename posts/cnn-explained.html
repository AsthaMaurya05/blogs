BLOG 02 · CONVOLUTIONAL NEURAL NETWORKS

A 3×3 grid of nine numbers, dragged across an image, is how a machine learns to see.

I went through 3Blue1Brown's convolution video, sat through MIT's 6.S191 CNN lecture twice, and read the full CS231n notes. This is what I wrote alongside all of that — what a filter actually computes, what "feature map" actually means, and the part that took me the longest to really get: why the same tiny window, repeated enough times and stacked deep enough, can eventually tell a cat from a car.

01 · The Problem That Started Everything

Why a plain neural network can't really see

Before CNNs existed, people tried to classify images by flattening them into a list of pixels and feeding that list into a regular fully-connected network. It mostly didn't work, and the reason is worth sitting with.

A modest 200×200 image in full colour isn't three numbers. It's 200 × 200 × 3 = 120,000 numbers. Feed those into the first hidden layer of a plain network, and a single neuron in that layer needs its own weight connecting to every one of those 120,000 inputs. Multiply that by the number of neurons in the layer — already in the millions of parameters for one layer alone. Training that becomes impossibly expensive. More importantly: even if it trained, it wouldn't generalize. A dog in the top-left corner of an image and the same dog in the bottom-right look completely different to a flat list of pixel values. The network would have to learn "dog" from scratch for every possible position. That's not learning what a dog looks like — it's just memorizing arrangements of pixels.

The key insight CNNs are built on Images have spatial structure. Nearby pixels are related. A dog's ear is made of adjacent pixels. A wheel exists in a local patch of an image. You don't need to compare pixel 1 with pixel 100,000 to understand local structure. CNNs exploit this by applying small filters that look at local regions — not the whole image at once — and reusing those same filters everywhere in the image.

This is the "X or X?" problem MIT's 6.S191 lecture opens with. Show a network two images of the letter X — one centered, one shifted slightly to the right — and both are numerically almost entirely different pixel matrices. A flat neural network has no reason to think they're the same thing. A CNN finds the same diagonal strokes regardless of where in the image they appear, because it looks for them locally, everywhere.

The three principles MIT's lecture distills this down to are worth keeping in one place: (1) apply a filter to extract local features, (2) use multiple filters to extract different features, (3) spatially share the parameters of each filter. Everything else in a CNN is an elaboration of those three ideas.

02 · The Core Operation

What a convolution actually is

The word sounds intimidating. The operation itself is one of the simplest things in deep learning — element-wise multiply, then add up. That's it.

A filter (also called a kernel) is a small grid of numbers, typically 3×3 or 5×5. You lay it on top of a patch of the image the same size, multiply each filter value by the pixel it's sitting on, and add all those products together. That sum becomes one value in the output — one pixel in what's called the feature map. Then the filter slides one step sideways (or by the stride amount), and you do it again. And again. Until the filter has covered the whole image.

For a 3×3 filter \(F\) applied to a patch of image \(X\): \[ \text{output} = \sum_{i=0}^{2} \sum_{j=0}^{2} X_{ij} \cdot F_{ij} \] Element-wise multiply every corresponding pair, then add all 9 results together. That's one value in the feature map.

Three parameters that control everything

Stride is how many pixels the filter moves each step. Stride 1 means dense overlap — the filter shifts by one pixel at a time, producing a large output. Stride 2 halves the output dimensions. It's the difference between "I want to look at everything closely" and "I'm willing to skip some overlap to save compute."

Padding decides what happens at the edges. Without padding, a 3×3 filter applied to a 5×5 image can only be placed where a full 3×3 patch fits — meaning the output shrinks to 3×3. Add a border of zeros around the input ("same" padding) and the output stays 5×5. Most modern implementations use same padding so spatial dimensions don't quietly shrink at every layer.

The output size formula, straight from CS231n:

\[ \text{output size} = \left\lfloor \frac{W - F + 2P}{S} \right\rfloor + 1 \] where \(W\) = input width, \(F\) = filter size, \(P\) = padding, \(S\) = stride.

Quick check: 7×7 input, 3×3 filter, stride 1, padding 0 → (7 − 3 + 0)/1 + 1 = 5. Matches what you'd expect — the filter has five positions horizontally and five vertically. With padding 1: (7 − 3 + 2)/1 + 1 = 7. Same size in, same size out.

Parameter sharing — the real efficiency win One filter has, say, 3×3 = 9 weights (for a grayscale image). Those same 9 weights are used at every position in the image. This is called parameter sharing, and it's what makes CNNs affordable. As CS231n puts it: if detecting a horizontal edge is useful at one location, it's probably useful everywhere else too. No need to relearn it for each of the 55×55 positions in a feature map — one filter, learned once, applied everywhere.
03 · What the Filters Actually Learn

You don't design the filters. The network does.

Here's the part that surprises people: you don't write a filter that detects edges, or one that detects fur, or one that detects wheels. You initialize them randomly, run backpropagation, and they learn to detect things on their own.

The MIT lecture makes this vivid with an example that's worth working through explicitly. Consider a filter designed to detect the top-left diagonal of an X. It might look like this — a 3×3 grid of 1s and -1s, with 1s where the diagonal stroke should be and -1s everywhere else. When this filter slides over a region of the image that actually contains that diagonal, the dot product produces a high positive value. When it slides over background, the 1s land on -1 pixels and the -1s land on 1 pixels, and everything cancels out to something near zero. High activation means "pattern found here." Low activation means "pattern not here."

The Lenna image from the lecture makes this visceral. Feed the original photo through an edge-detection filter and you get an image where the bright areas are exactly where sharp edges are — the outline of the hat, the curve of the shoulder, the border between light and shadow. A filter with just nine values in it, applied everywhere, extracts an entire structural sketch of the image. And crucially: a CNN doesn't hardcode these filters. It starts with random values and learns through gradient descent which filters minimize the loss. Some of AlexNet's 96 learned 11×11 filters look like edge detectors. Some look like blob detectors. Some detect colour gradients. They all emerged from random initialization and lots of gradient updates.

What CS231n says about the filters AlexNet learned The 96 learned filters from Krizhevsky et al. (AlexNet, 2012) are each size 11×11×3 and are shared by the 55×55 neurons in one depth slice. The pattern makes intuitive sense: if detecting a horizontal edge is useful at one location, it should be useful at all other locations too — so there's no reason to relearn it 55×55 times. One filter, learned once, applied everywhere. That's parameter sharing in action.

Multiple filters, multiple feature maps

You don't apply just one filter. You apply, say, 32 of them. Each one learns to detect something different. Each one produces its own feature map — a spatial map of "where in the image was this particular pattern present, and how strongly?" Stack those 32 feature maps and the output of a convolutional layer has shape H × W × 32. This is what CS231n calls the output volume — 3D, not 2D. Width and height track spatial location. Depth (number of filters) tracks which feature we're talking about.

A single neuron in the first conv layer is only connected to a small 3×3 patch of the input. But a neuron in a later conv layer is connected to a region that maps back to a much larger area of the original image — its receptive field grows with each layer. Early layers look at tiny patches. Late layers, through the stacking, effectively "see" large regions of the original image.

04 · Introducing Non-Linearity

ReLU — applying it after every convolution, always

Convolution is a linear operation. Stack two linear operations and you get one linear operation — all that depth buys you nothing. ReLU is what breaks the linearity and lets the network learn anything complicated.

The operation itself is the definition of simple: \(\text{ReLU}(z) = \max(0, z)\). Every negative value in the feature map becomes zero. Every positive value stays exactly as it is. That's it. No curve, no exponential. Applied pixel-by-pixel across the entire feature map.

The MIT lecture shows this visually with a building photograph. Before ReLU: a grainy feature map where black pixels represent negative activations and white represents positive. After ReLU: the same map, but all the black is gone — only the positive activations remain. The structure you care about (edges, textures) was already mostly in the positive values. The negative values were suppressing nothing important, and removing them keeps the signal clean.

Why not sigmoid? We went through this in the neural networks post. Sigmoid saturates — for large positive or negative inputs, its derivative is nearly zero. Chain a dozen of those derivatives together through backprop and the gradient reaching the early layers is essentially nothing, so those layers stop learning. ReLU's derivative is either 0 (for negative inputs) or a flat, constant 1 (for positive ones). A gradient passing through an active ReLU neuron comes out the other side exactly as strong as it went in. That's the whole reason deep networks became trainable.
05 · Throwing Away the Right Information

Pooling — compress without losing what matters

After convolution and ReLU, the feature maps are still large. Pooling shrinks them. The specific choice of which value to keep — the maximum — is what makes this smarter than just throwing pixels away.

Max pooling divides the feature map into non-overlapping 2×2 windows and keeps the highest value from each. The exact numbers from MIT's lecture: input is 1,1,2,4 / 5,6,7,8 / 3,2,1,0 / 1,2,3,4. After 2×2 max pool with stride 2, output is 6,8 / 3,4. Four values collapse to one, and the maximum — the strongest detection signal — survives.

Why max, specifically? The maximum activation in a region is evidence that a feature was detected somewhere in that region. You don't always care exactly where — a nose slightly left or right of center doesn't change the fact that it's a face. By keeping the maximum, you retain "yes, this feature was found here" while discarding the precise pixel-level address. That's translation invariance. Small shifts in the input don't change the output.

Getting rid of pooling entirely CS231n notes that "many people dislike the pooling operation." Some modern architectures — particularly those for generative models like VAEs and GANs — drop pooling and use strided convolutions to downsample instead. The trend in vision transformers is also moving away from explicit pooling. But for classification CNNs, max pooling is still the default and shows up in essentially every interview question on the topic.
06 · Putting It All Together

How the layers stack — and what each stage actually sees

Each individual layer does something simple. The interesting behaviour emerges from stacking them. CS231n traces this precisely for CIFAR-10 and it's worth following the dimensions all the way through.

Start with 32×32×3 — width × height × RGB channels. Run through a conv layer with 12 filters of size 3×3, stride 1, same padding: output is 32×32×12. ReLU leaves shape unchanged. Pooling (2×2, stride 2) halves both spatial dims: 16×16×12. Stack another conv+relu+pool block: 8×8×24. Flatten to 8×8×24 = 1,536 values, feed into FC(128) then FC(10) with softmax. Done.

\[ [32 \times 32 \times 3] \xrightarrow{\text{CONV}} [32 \times 32 \times 12] \xrightarrow{\text{POOL}} [16 \times 16 \times 12] \xrightarrow{\text{CONV}} [16 \times 16 \times 24] \xrightarrow{\text{POOL}} [8 \times 8 \times 24] \xrightarrow{\text{Flatten + FC}} [10] \]

The pattern: spatial size shrinks, depth grows. Late conv layers look at small spatial patches that map back to large regions of the original image — their receptive field is large even though the filter itself is still 3×3. Early layers see tiny local patches. Late layers, through stacking, effectively see the whole image.

07 · From Theory To Actually Running Code

What this looks like when you sit down to build one

The MIT 6.S191 lecture slides show both Keras and PyTorch APIs side by side. Here's what the standard architecture looks like in Keras, with every argument annotated so you actually understand what you typed.

Dropout at 0.5 means 50% of FC neurons are randomly zeroed during each training step. This stops the network from memorising specific pathways and forces it to build redundancy. At inference time, dropout switches off. It costs nothing at prediction time and consistently reduces overfitting. Use it.

The training loop underneath all of this

The weights start random. Then, for each batch of images:

  1. Forward pass — input flows through every layer to produce a prediction vector.
  2. Loss — compare prediction against ground truth using cross-entropy: \(L = -\sum y_i \log(p_i)\). If the correct class gets probability 0.94, loss is low. If it gets 0.02, loss is high.
  3. Backward pass — gradients of the loss flow back through every layer via the chain rule, telling each filter weight which direction to move to reduce the loss.
  4. Update — Adam optimizer adjusts all weights by a small amount in the direction that reduces loss. Repeat for every batch, every epoch.

After thousands of batches, the filters that started random have settled into edge detectors, texture detectors, shape detectors — whatever the data required. Nobody programmed those features in. They fell out of minimizing the loss.

08 · The Practical Thing You'll Actually Use

Transfer learning — don't train from scratch

Training a CNN from scratch on a new dataset is almost never the right move for a project. Someone has already trained one on 14 million images and published the weights. Starting there is faster, more accurate, and requires far less data.

The intuition is grounded in the feature hierarchy. A network trained on ImageNet already knows how to detect edges, textures, and object parts. Those filters don't become useless when you want to classify something new — they're generally useful for any natural image task. So: take the pretrained network, freeze those weights, attach a new classification head for your classes, and only train the head.

GlobalAveragePooling2D averages each feature map to a single value. If EfficientNet's last conv block outputs 7×7×1280, GAP collapses it to a 1280-element vector — one number per filter, representing the average presence of that feature across the image. Cleaner than flattening (which would give 7×7×1280 = 62,720 values) and more robust to small spatial shifts.

09 · A Reality Check Worth Sitting With

What CNNs actually learn vs. what we say they learn

The neat story — early layers detect edges, deep layers detect objects — is broadly true. It's also a simplification that breaks in interesting ways.

CS231n notes that AlexNet's first-layer filters do look like edge detectors and colour blob detectors — the visualizations are striking. But go deeper and the feature maps resist human interpretation. A neuron in layer 4 doesn't respond to "a wheel" — it responds to some activation pattern from the previous layer that happened to be useful for distinguishing cars. We call that pattern "wheel-like" because it's the nearest human concept. The network has no concept of a wheel.

The clearest demonstration of this gap is adversarial examples. Take an image a network classifies as a cat with 99% confidence. Add a tiny, carefully-constructed pixel perturbation — invisible to a human — and it suddenly outputs "guacamole" with 87% confidence. The learned structure is real enough to classify cats correctly in normal conditions, but it's not the same structure humans use to recognize them. It found a different, more brittle path to the same answer.

Why this matters for your projects This is why model interpretability tools like Grad-CAM exist. Grad-CAM highlights which parts of the input image most activated the prediction — giving you a heatmap of "where the network was looking." For a placement project, adding Grad-CAM visualization on top of an EfficientNet classifier transforms it from "a thing that predicts labels" into "a thing that shows its reasoning." That's the difference between a demo and a portfolio piece.

If you forget everything else, keep these

  • Output size formula: (W − F + 2P) / S + 1. Know this without thinking.
  • Filters double as you go deeper (32→64→128) while spatial size halves via pooling. Trade spatial resolution for feature richness.
  • Same padding at every conv so spatial dims only shrink at pooling layers, deliberately.
  • ReLU after every conv, always. Stacking linear operations without non-linearity collapses to one linear operation.
  • The filters are not designed. They start random and learn through backprop. The edge-detector story is what they end up looking like, not what they were initialised to be.
  • Transfer learning first. Pretrained EfficientNetB0 will beat your from-scratch network on almost any dataset under 100K images.
  • Receptive field grows with depth. A 3×3 filter in layer 5 effectively sees a much larger region of the original image than a 3×3 filter in layer 1.
